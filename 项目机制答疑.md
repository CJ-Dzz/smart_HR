# Smart-HR 机制答疑

本文回答你提出的 3 个问题，并结合当前代码实现说明。

## 1) 每个部分的 Prompt 在哪？每个阶段内容是什么？

### A. 简历技能提取阶段（上传简历时）
- 入口：
  - `back/src/main/java/com/smarthr/service/hr/ResumeService.java` -> `uploadResume` -> `skillExtractor.extractSkills(...)`
- Prompt 位置：
  - `back/src/main/java/com/smarthr/service/document/SkillExtractor.java` -> `extractWithAI`
- 阶段内容：
  1. 先走知识图谱关键词匹配（`extractFromKnowledgeGraph`）
  2. 再用 AI 补充技能（`extractWithAI`）
  3. 将两者合并去重
- Prompt 特征：
  - 单轮 `user` 消息
  - 要求“只返回技能名列表，用逗号分隔”
  - 输入会截断（`truncateContent(content, 2000)`）

### B. 面试题生成阶段（面试官侧）
- 入口：
  - `back/src/main/java/com/smarthr/service/interview/InterviewQuestionService.java` -> `generateWithAI`
- Prompt 位置：
  - `buildPrompt(...)`（同文件）
  - 题库检索上下文由 `QuestionBankService.formatContext(...)` 提供
- 阶段内容：
  1. 先从题库向量检索相似题（Milvus）
  2. 拼接“技能 + 业务域 + 命中题库上下文 + 输出 JSON 约束”Prompt
  3. 以 `system + user` 两段消息调用模型
  4. 解析 JSON；失败则降级默认题目（`generateDefaultQuestions`）
- Prompt 特征：
  - 明确要求仅输出 JSON 数组
  - 强调业务域和公司语境（Smart-HR/企业金融支付）

### C. HR 综合匹配分析阶段（简历-岗位匹配）
- 入口：
  - `back/src/main/java/com/smarthr/service/hr/MatchService.java` -> `match(...)`
  - 调用 `HybridMatchService.match(...)`
- Prompt 位置：
  - `back/src/main/java/com/smarthr/service/match/HybridMatchService.java`
  - `evaluateWithLLM(...)` + `buildLLMPrompt(...)`
- 阶段内容：
  1. 先做知识图谱技能匹配（graph score）
  2. 将 JD、简历、匹配/缺失技能、技能分等喂给 LLM
  3. 解析 LLM 输出中的“评分/评估报告”
  4. 按 `Graph(0.5) + LLM(0.5)` 计算最终分
- Prompt 特征：
  - 有固定 system 角色（HR 招聘专家）
  - user prompt 包含结构化字段和返回格式要求
  - LLM失败时有兜底：`score=70` + 固定降级文案

## 2) 知识图谱是如何使用和更新的？

### 如何使用
- 图数据存储：Neo4j
- 节点/关系：
  - 节点：`Skill`、`SkillCategory`
  - 关系：`REQUIRES`、`BELONGS_TO`、`RELATED_TO`
- 主要使用位置：
  - `SkillGraphService.extractSkills(...)`：从文本抽技能
  - `SkillGraphService.calculateSkillMatch(...)`：算匹配分
  - `SkillRepository` 中 Cypher 查询：前置技能/相关技能/高级技能/依赖树等
- 在匹配中的作用：
  - 匹配分不是纯关键字相等，还会根据依赖和相关性给“部分匹配分”

### 如何更新
- 当前项目里图谱主要是“离线初始化/扩容”，不是在线自动学习。
- 更新方式：
  1. 修改或新增 Cypher 脚本：
     - `docker/neo4j/init.cypher`
     - `docker/neo4j/init-skills-extended.cypher`
  2. 通过 `cypher-shell` 或 Neo4j Browser 执行脚本导入
- 代码层面没有提供“后台管理接口”去动态增删图谱节点；你要更新技能体系，主要改脚本并重导。

## 3) RAG 在项目中怎么用？为什么有的地方用、有的地方没用？

### 已使用 RAG/向量检索的地方
- 面试题生成（Interview）：
  - `QuestionBankService.searchSimilarQuestions(...)`
  - 流程：查询文本 -> embedding -> Milvus 题库检索 -> 把命中结果注入 Prompt
  - 目标：让题目更贴合企业语境和历史题库

- 简历向量化入库：
  - `ResumeService.indexResume(...)`
  - 流程：简历内容 embedding -> 写入 `MilvusVectorStore`
  - 作用：为后续语义检索能力做数据准备

### 明确禁用 RAG 的地方
- HR 综合匹配评分：
  - `HybridMatchService` 中 RAG 逻辑被注释/置零
  - `ragScore` 固定 `0.0`
  - 最终分只用 `Graph + LLM`
- 代码里有明确注释：RAG 已禁用，保留字段为后续可能重启用。

### 为什么会“有的用、有的不用”
- 从当前实现看是“业务策略分流”：
  1. 面试题生成：RAG价值高（需要参考题库上下文）
  2. HR匹配评分：当前版本强调“知识图谱 + LLM”双维度，刻意关闭 RAG 权重，降低复杂度与不稳定因素
  3. 简历向量入库先保留，为后续恢复“语义检索加权”留扩展位

## 你后续要改这三块时的建议入口
- 想改 Prompt：优先看
  - `SkillExtractor.extractWithAI`
  - `InterviewQuestionService.buildPrompt`
  - `HybridMatchService.buildLLMPrompt`
- 想改图谱：优先改 `docker/neo4j/*.cypher`
- 想恢复匹配 RAG：从 `HybridMatchService` 的注释代码恢复 `calculateRAGScore`，并调整权重/DTO 展示。
